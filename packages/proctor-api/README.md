# proctor-api

Game orchestrator and GraphQL API for Arena. Manages game sessions, drives AI agent turns via LLM calls, and streams render instructions to the front-end over SSE. Currently runs poker; the game engine layer is pluggable.

## Quick Start

```sh
npm run proctor-api          # dev server (tsx watch, port 4001)
npm run test:proctor-api     # vitest
npm run codegen --workspace=@arena/proctor-api  # regenerate resolverTypes.ts
npx tsc --noEmit -p packages/proctor-api        # type check (from repo root)
```

Environment variables come from the root `.env` file (loaded via `dotenv` with a relative path). See `.env.example` at the repo root.

## Architecture

### GraphQL Layer (`src/gql/`)

- Schema uses SDL in TypeScript template literals tagged with `/* GraphQL */`
- One folder per entity under `src/gql/schema/` (e.g. `Channel/`, `Game/`, `GameState/`, `Hand/`, `Player/`, `RenderInstruction/`), each with `typeDefs.ts` and `resolvers.ts`
- Merged via `@graphql-tools/merge` in `mergedTypeDefs.ts` / `mergedResolvers.ts`
- Types generated by `@graphql-codegen` into `resolverTypes.ts` (checked in, excluded from Biome)
- Resolver typing: use `QueryResolvers["fieldName"]` and `MutationResolvers["fieldName"]` from `resolverTypes.ts`
- Server: `graphql-yoga` with SSE subscriptions for streaming render instructions

### Session Management (`src/services/session/`)

- **session-manager.ts** -- in-memory session store; create/get/stop sessions, handle instruction ACKs
- **programming.ts** -- programming loop: runs game modules in sequence while the channel is live, handles crash recovery via DynamoDB snapshots
- **pubsub.ts** -- in-process pub/sub for SSE instruction streaming
- **ack-gate.ts** -- Promise-based gate that blocks the orchestrator until the front-end ACKs specific instructions

### Poker Game Engine (`src/services/games/poker/`)

Two layers:

1. **poker-engine/** -- Pure rules engine wrapping `poker-ts`. State management, legal actions, hand evaluation. No I/O.
2. **orchestrator/** -- Game loop that drives hands, calls agents, and emits render instructions.
   - `session-loop.ts` -- top-level session runner (manages hands, blinds, eliminations)
   - `hand-loop.ts` -- runs a single hand through betting rounds
   - `turn-resolver.ts` -- resolves a player turn (calls agent runner, validates, retries)
   - `emitter.ts` -- emits render instructions via pub/sub

Supporting modules:
- **agent-runner.ts** -- `AgentRunner` interface (init, runTurn, rejectAction, promptReaction)
- **llm-agent-runner.ts** -- LLM implementation using Vercel AI SDK (multi-provider: Anthropic, OpenAI, Google, xAI, DeepSeek, Bedrock)
- **instruction-builder.ts** -- builds typed `RenderInstruction` payloads
- **message-formatter.ts** -- human-readable game event strings for agent conversations
- **prompt-template.ts** -- builds per-agent system prompts from template + persona
- **prompts/** -- `system.ts` (system prompt template), `hand-reaction.ts` (post-hand reaction prompt), `personas/` (per-persona strategy and commentary)
- **fallback-lines.ts** -- humorous fallback commentary for auto-check/fold on LLM failure

### Other Source Files

- **game-config.ts** -- `GameConfig` / `AgentConfig` interfaces, `createGameConfig()` function, blind schedule
- **characters.ts** -- character roster (persona, bio, voiceDirective, ttsVoice per character)
- **logger.ts** -- error logging to file + console
- **types.ts** -- re-exports types from `resolverTypes.ts` plus internal `CreateGameOptions`

### Persistence (`src/persistence.ts`, `src/db.ts`)

DynamoDB via `@aws-sdk/lib-dynamodb`. Tables: modules, instructions, channel-state, settings, agent-messages. Table names are prefixed with `TABLE_PREFIX` env var (default `arena-`).

## Testing

Tests use vitest. Test files mirror the source structure under `test/`.

- Integration tests use `yoga.fetch()` via `test/yoga-helper.ts` (no HTTP server needed)
- DynamoDB persistence is mocked with `vi.mock("../src/persistence.js", ...)`
- Call `_resetSessions()`, `_resetPubSub()`, `_resetGames()`, and `_resetAckGate()` in `beforeEach` for isolation
- `test/fixtures/scripted-agent.ts` provides a deterministic `AgentRunner` for orchestrator tests

## Adding a New Game Engine

1. Create `src/services/games/<game>/` with a rules engine and orchestrator
2. Implement the `AgentRunner` interface for agent interaction
3. Add the game type to the `PROGRAMMING` array in `src/services/session/programming.ts`
4. Add GraphQL schema entities as needed under `src/gql/schema/`
