# @arena/front-end

Pure renderer for the Arena poker stream. Subscribes to SSE render instructions from `proctor-api`, renders them with animations and TTS, and ACKs each one. Contains zero game logic -- all game state comes from the server.

## Quick Start

```sh
# From monorepo root
npm run front-end          # Vite dev server (proxies /graphql to localhost:4001)
npm run test:front-end     # Run tests
npm run codegen:front-end  # Regenerate GraphQL types from proctor-api schema
npx tsc --noEmit -p packages/front-end  # Type check
```

Env vars are read from the root `.env` file (`envDir: "../../"` in vite.config.ts). Only `VITE_`-prefixed and `INWORLD_`-prefixed vars are exposed to client code. Set `VITE_DISABLE_TTS=true` to silence TTS during development.

## Architecture

### Data Flow

```
SSE stream --> RenderQueue (animation/TTS gates) --> useReducer dispatch --> React
                                                          |
                                                    GameState.currentView
                                                          |
                                                    useRouteSync --> navigate()
```

### Instruction-Driven Rendering

The proctor sends typed render instructions over SSE (GAME_START, DEAL_HANDS, PLAYER_TURN, etc.). The front-end never initiates game actions -- it only renders what it receives and ACKs after processing. Navigation between pages (`/poker`, `/poker/endcard`) is also instruction-driven via the `currentView` field in GameState.

### Render Queue

The render queue (`session/renderQueue.ts`) decouples instruction arrival from rendering. Each instruction type has a handler with optional `preWait`/`postWait` hooks for animation timing and TTS coordination. The queue uses Promise-based gates so deal animations, display pauses, and TTS playback complete before the next instruction renders. ACKs fire after each instruction is fully processed.

### State Management

A single `useReducer` holds all game state (`GameState` in `types.ts`). The reducer delegates to per-instruction-type handler functions in `session/handlers/`. Mappers in `session/mappers.ts` translate GraphQL types to app types. State is immutable -- handlers return new objects.

### TTS

Streaming TTS via Inworld (`inworld-tts-1.5-mini`), with raw PCM piped to Web Audio API for low-latency playback. Voice IDs are stored per-player from GAME_START metadata. TTS is coordinated with the render queue so speech completes before the next instruction.

### SSE and Reconnection

`useSSELoop` manages the SSE lifecycle: subscribe, push instructions to the render queue, ACK via GraphQL mutation. On disconnect, it fetches current channel state via a `connect` query, dispatches a RECONNECT action to restore state, and re-subscribes.

## Key Conventions

- **CSS Modules** for component styling (`.module.css` files colocated with components).
- **GraphQL operations** are raw SDL strings with `/* GraphQL */` tags in `graphql/operations.ts`. Types are generated by `@graphql-codegen`.
- **Animation timing** constants live in `session/timing.ts` as the single source of truth shared by the render queue and animation hooks.
- **Mock routes** (`/poker/mock`, `/poker/endcard/mock`) render with fixture data for visual development without a running proctor.
- **Components are props-driven** -- pages receive all data as props from `App.tsx`, which reads from the reducer state.
- **No direct game logic** -- the front-end never computes hand rankings, pot splits, or betting rules.

## Testing

Tests live in `test/` and use vitest. Current coverage includes chip formatting, mappers (GraphQL to app types), and the reducer (instruction handling, state transitions).

```sh
npm run test:front-end
```
