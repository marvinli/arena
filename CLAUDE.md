# Arena

AI agents compete in games on a live stream. Everything is TypeScript.

## Architecture

See [docs/HIGH_LEVEL_DESIGN.md](docs/HIGH_LEVEL_DESIGN.md) for the full system design.

## Packages

- `proctor-api` — orchestrator + game engines (GraphQL, port 4001). Business logic under `src/services/` (session management at `services/session/`, poker at `services/games/poker/`)
- `front-end` — React renderer (TTS, animations, no game logic)
- `videographer` — headless browser capture/streaming (planned, not yet implemented)

## Monorepo Setup

- npm workspaces at `packages/*`
- Root tsconfig: ES2022, Node16, strict
- All packages are ESM (`"type": "module"`)
- Biome for linting/formatting (`npm run check`, `npm run check:fix`), configured globally at root `biome.json`
- `resolverTypes.ts` is generated but checked in (not gitignored) — excluded from Biome

## Game Configuration

Game config (players, blinds, chips, hands per game) lives server-side in `packages/proctor-api/src/game-config.ts`. The `startSession` mutation takes only a `channelKey` — config is not passed from the front-end.

Each agent is configured with: `playerId`, `name`, `modelId`, `modelName`, `provider`, `avatarUrl`, `ttsVoice`, `temperature`.

## Key Dependencies

- `poker-ts` — poker rules engine (wraps `Poker.Table`)
- `ai` (Vercel AI SDK) — multi-provider LLM calls for agents (Anthropic, OpenAI, Google, xAI, DeepSeek, Amazon Bedrock)
- `graphql-yoga` — GraphQL server with SSE subscriptions
- `better-sqlite3` — local SQLite database
- `@lobehub/icons` — AI provider brand icons for the front-end

## Conventions

- GraphQL schema: SDL in TypeScript template literals with `/* GraphQL */` tag, one folder per entity under `src/gql/schema/` (typeDefs.ts + resolvers.ts), merged via `@graphql-tools/merge`
- Resolver types generated by `@graphql-codegen` — use `QueryResolvers["fieldName"]` and `MutationResolvers["fieldName"]` for typing individual resolvers
- Types: prefer re-exporting from generated `resolverTypes.ts` rather than hand-writing duplicates
- Business logic lives under `src/services/` — session management at `services/session/`, game engines at `services/games/<game>/` (e.g., `services/games/poker/`)
- Tests: vitest, `_resetGames()` + `_resetSessions()` + `_resetPubSub()` in `beforeEach` for isolation. Integration tests use `yoga.fetch()` via `test/yoga-helper.ts`
- No enum resolvers needed — poker-engine returns generated enum values (e.g. `PlayerStatus.Active` = `"ACTIVE"`) which match GraphQL enum names directly

## proctor-api Structure

```
src/services/games/poker/
  poker-engine/          # Pure rules engine (state, actions, hand evaluation)
    state.ts, actions.ts, hand.ts, store.ts, types.ts, index.ts
  orchestrator/          # Game loop — drives hands, calls agents, emits instructions
    session-loop.ts      # Top-level session runner
    hand-loop.ts         # Runs a single hand
    turn-resolver.ts     # Resolves player turns (calls agent runner)
    emitter.ts           # Emits render instructions to front-end
    types.ts, index.ts
  agent-runner.ts        # AgentRunner interface + context/result types
  llm-agent-runner.ts    # LLM implementation (ai SDK, multi-provider)
  instruction-builder.ts # Builds poker-specific RenderInstruction payloads
  message-formatter.ts   # Human-readable game event strings for agent conversations
  prompt-template.ts     # Shared system prompt template for agents
  fallback-lines.ts      # Fallback commentary when agent analysis fails
```

## Front-End Architecture

The front-end is a pure renderer — no game logic. It subscribes to SSE render instructions, renders them with animations and TTS, and acks each one.

```
src/
  session/               # State management
    reducer.ts           # Dispatch table — delegates to handlers/
    handlers/            # Per-instruction-type state handlers
    renderQueue.ts       # Animation gate + TTS coordination
    timing.ts            # Shared animation timing constants
    sseClient.ts         # SSE subscription + channel state fetching
    config.ts            # Channel key + delay helper
    mappers.ts           # GraphQL → app type mappers
    types.ts             # GQL type aliases
  hooks/
    useGameSession.ts    # Session lifecycle (start/stop)
    useSSELoop.ts        # SSE subscribe/ack/reconnect loop
    useDealAnimation.ts  # Staged hole card deal animation
    useCommunityDealAnimation.ts
  components/
    PokerTable.tsx       # Main table composition
    CommunityArea.tsx    # Pot labels + 5 card slots
    BetIndicator.tsx     # Bet chip with toward-center positioning
    SidePanel.tsx        # Speaking player + analysis text
    PlayerSeat.tsx       # Individual player seat
    PlayingCard.tsx      # Card with 3D flip animation
    ChipStack.tsx        # Chip icon + formatted amount
    ProviderIcon.tsx     # AI provider brand icons
    layout.ts            # Seat positions + colors (data, not UI)
  tts.ts                 # OpenAI streaming TTS (gpt-4o-mini-tts, PCM via Web Audio API)
  types.ts               # Core app types (Card, Player, GameState, etc.)
```

### Data Flow

```
SSE stream → RenderQueue (animation/TTS gates) → useReducer dispatch → React components
```

The render queue decouples instruction arrival rate from rendering. It uses Promise-based gates to sequence deal animations, display pauses (showdown, leaderboard), and TTS playback. The front-end ACKs each instruction immediately via GraphQL mutation, enabling the proctor to pipeline the next LLM call while the front-end renders at its own pace.

### TTS

Uses OpenAI streaming TTS (`gpt-4o-mini-tts`) with raw PCM output piped directly to Web Audio API for low-latency playback. Voice IDs are stored per-player from `GAME_START` `playerMeta`.
